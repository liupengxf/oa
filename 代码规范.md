## 代码规范

### 1 规范原则

	1. 一致性：全项目编码风格、命名规则、文件结构保持统一；
	2. 可维护性：兼顾功能完整性与后续调试、扩展。

### 2 后端规范

1. 项目包名为com.buu.oa

2. MyBatis-Plus要求使用Mapper 接口 + XML 分离模式

3. mapper模版要求如下格式：

   ~~~xml
   <?xml version="1.0" encoding="UTF-8"?>
   <!DOCTYPE mapper PUBLIC "-//mybatis.org//DTD Mapper 3.0//EN"
           "http://mybatis.org/dtd/mybatis-3-mapper.dtd">
   <mapper namespace="">
       
   </mapper>
   ~~~

4. SQL语句中可能出现语法问题的部分，必须使用转义字符

5. 复杂 SQL（如多条件筛选、联表查询）必须添加注释说明业务逻辑

6. 项目使用controller-service-mapper-entity分层开发

7. 采用统一返回

8. application配置文件使用properties格式，并且提供两个版本，一个是中文注释版本，一个是简单易懂的英文注释版本

9. 要求生成时提供完整的后端代码，如果可以复用之前代码，则复用。如果不能复用，则生成新的代码。如果需要对已有代码进行改，提供修改方案，并且提供完整的修改之后的代码。

10. 生成的代码提供完整的存储路径

### 3 前端规范

1. 开发时使用index.html作为主页面
2. 所有功能尽量单独形成页面，最终集成到index.html中
3. 要求页面与提供的预览保持风格或者形态一致，如果没有提供预览，则与提供代码中的页面保持一致
4. 提供的前端代码提供命名和放置的位置

### 4 注释规范

1. **类注释**：所有类（实体 / Service/Controller）必须添加类注释，说明功能，示例：

```java
/**
 * 工单Controller
 * 处理工单相关前端请求，包含查询、创建、状态更新等接口
 */
@RestController
@RequestMapping("/api/work-order")
public class WorkOrderController { ... }
```

2. **方法注释**：Service/Controller 中公共方法必须添加注释，说明参数、返回值、异常，示例：

```java
/**
 * 工单状态更新
 * @param orderId 工单ID
 * @param newStatus 新状态（PENDING/ASSIGNED等）
 * @return 更新结果
 * @throws StateTransitionException 状态流转非法时抛出
 */
public boolean updateOrderStatus(Long orderId, String newStatus) { ... }
```

3. **关键业务逻辑注释**：状态机校验、派单算法等核心逻辑，需在代码行旁添加单行注释。

4. **前端组件注释**：Vue 组件顶部添加注释，说明组件功能、Props、事件，示例：

   ```vue
   <template>
     <!-- 工单状态选择器 -->
     <el-select v-model="status" placeholder="请选择工单状态">
       ...
     </el-select>
   </template>
   <script setup>
   /**
    * 工单状态选择器组件
    * @props {String} value - 绑定的状态值
    * @emits {String} change - 状态变更时触发
    */
   ...
   </script>
   ```

   5. **工具函数注释**：所有工具函数必须添加 JSDoc 注释，说明参数、返回值。
